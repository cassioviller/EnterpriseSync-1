# PROMPT 3: Gestão de Restaurantes e Marmitas (V2)

**Objetivo:** Aprimorar o módulo de Alimentação existente para permitir o lançamento detalhado de marmitas/refeições por funcionário e por centro de custo, suportando múltiplos itens no mesmo lançamento. Além disso, integrar automaticamente com a nova Gestão de Custos (Prompt 2).

**Arquivos a serem modificados:**
- `models.py`
- `alimentacao_views.py`
- `templates/alimentacao/novo_lancamento.html` (ou equivalente)
- `utils/financeiro_integration.py` (chamada da função `registrar_custo_automatico`)

Por favor, implemente as seguintes alterações passo a passo:

## Passo 1: Atualização do Modelo AlimentacaoLancamentoItem
Atualmente, o sistema possui `AlimentacaoLancamento` e `AlimentacaoLancamentoItem`. Precisamos descer o nível de detalhe para saber *quem* comeu e para *qual centro de custo* a despesa vai.
No arquivo `models.py`, atualize a classe `AlimentacaoLancamentoItem`:
```python
# Adicionar na classe AlimentacaoLancamentoItem
funcionario_id = db.Column(db.Integer, db.ForeignKey('funcionario.id'), nullable=True)
centro_custo_id = db.Column(db.Integer, db.ForeignKey('centro_custo.id'), nullable=True)

# Relacionamentos
funcionario = db.relationship('Funcionario', backref='refeicoes_consumidas')
centro_custo = db.relationship('CentroCusto', backref='despesas_alimentacao')
```

## Passo 2: Interface de Lançamento de Marmitas
No template de lançamento de alimentação (`templates/alimentacao/novo_lancamento.html`):
1. O cabeçalho do formulário deve pedir: Restaurante, Data e Obra (opcional, para preencher o centro de custo padrão).
2. Crie uma tabela dinâmica (usando JavaScript) para adicionar múltiplos itens (marmitas).
3. Cada linha da tabela deve ter:
   - **Item:** Select com os itens do restaurante (ex: Marmita G, Marmita P, Refrigerante).
   - **Funcionário:** Select com a lista de funcionários ativos (com opção "Sem vínculo" para visitantes).
   - **Centro de Custo:** Select com os centros de custo (puxa o da obra por padrão, mas permite alterar).
   - **Quantidade:** Input numérico (default 1).
   - **Preço Unitário:** Preenchido automaticamente ao escolher o item.
   - **Subtotal:** Calculado via JS.
4. Um botão "Adicionar Linha" para inserir mais refeições no mesmo pedido.

## Passo 3: Processamento no Backend (alimentacao_views.py)
Na rota que salva o lançamento:
1. Crie o `AlimentacaoLancamento` principal (cabeçalho).
2. Faça um loop nos itens enviados pelo formulário dinâmico.
3. Para cada item, crie um `AlimentacaoLancamentoItem` vinculando o `funcionario_id` e o `centro_custo_id` informados na linha.
4. Calcule o `valor_total` do lançamento somando os subtotais.

## Passo 4: Integração Imediata com Gestão de Custos
Sempre que um lançamento de alimentação for salvo, o sistema deve gerar automaticamente os custos na Gestão de Custos (Prompt 2).
1. No backend, após salvar os itens, agrupe os valores por `restaurante_id`.
2. Para cada grupo (ou para o lançamento total), chame a função `registrar_custo_automatico` (criada no Prompt 2):
   - `tipo_categoria = 'ALIMENTACAO'`
   - `entidade_nome = restaurante.nome`
   - `entidade_id = restaurante.id`
   - `data = lancamento.data`
   - `descricao = f"Refeições - {len(itens)} itens"`
   - `valor = lancamento.valor_total`
   - `obra_id = lancamento.obra_id`
   - `origem_tabela = 'alimentacao_lancamento'`
   - `origem_id = lancamento.id`
3. Isso garante que, na tela de Gestão de Custos, apareça um Registro Pai "Despesa Alimentação - Restaurante X" com o valor total, e ao expandir, mostre os detalhes das marmitas.

## Passo 5: Proteção V1/V2
Envolva a exibição das colunas "Funcionário" e "Centro de Custo" no template com `{% if is_v2_active() %}` (conforme Prompt 0). Se for V1, o lançamento continua sendo genérico apenas por Obra. Se for V2, exige o detalhamento por funcionário/centro de custo.
