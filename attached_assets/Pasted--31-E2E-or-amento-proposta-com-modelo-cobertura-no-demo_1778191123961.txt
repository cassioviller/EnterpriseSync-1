#31 - E2E orçamento → proposta com modelo + cobertura no demo Alfa
What & Why
Diagnóstico read-only do demo Construtora Alfa (admin_id=17) no banco de produção mostrou o ciclo "modelo → orçamento → proposta" mal coberto:

0 templates de proposta cadastrados (nem do tenant, nem publico=True) → impossível testar o caminho com modelo manualmente; o <select name="template_id"> no modal "Gerar Proposta" aparece vazio.
Só 1 das 3 propostas (id=29, obra 25) tem orcamento_id preenchido. As outras 2 (28 e 30) foram criadas direto, sem orçamento prévio.
Nenhuma das 3 propostas tem proposta_template_id (todas NULL) nem cláusulas (n_clausulas=0).
Existe apenas 1 orçamento no demo, ainda em status rascunho.
O código de geração está correto (views/orcamentos_views.py:494‑696, Task #31): aceita template_id, copia cláusulas e campos do modelo, vincula orcamento_id, marca campos pendentes de revisão. O problema é exclusivamente de dados de demonstração + falta de teste automatizado E2E que cubra o caminho completo "criar template → criar orçamento → POST gerar-proposta com template_id".

Done looks like
O demo Alfa passa a ter pelo menos 1 PropostaTemplate completo, com cláusulas, condições, prazo, validade, itens inclusos/exclusos preenchidos. Marcado como padrao=True para aparecer pré-selecionado.
O demo Alfa passa a ter pelo menos 2 orçamentos (1 em rascunho, 1 fechado).
Pelo menos 2 das 3 propostas do demo passam a ter orcamento_id preenchido (vínculo orçamento→proposta visível na UI).
Pelo menos 1 proposta tem proposta_template_id preenchido, com cláusulas copiadas do template e campos_pendentes_revisao populado.
Existe um teste E2E automatizado (tests/test_e2e_orcamento_proposta_modelo.py ou nome similar) que:
Loga como admin Alfa.
Cria um PropostaTemplate via POST real.
Cria um Orçamento com pelo menos 2 itens via POST real.
POST /orcamentos/<id>/gerar-proposta passando template_id.
Verifica no banco: proposta.orcamento_id, proposta.proposta_template_id, proposta.prazo_entrega_dias (do template), proposta.itens_inclusos (do template), número de cláusulas copiadas, campos_pendentes_revisao não-vazio.
Verifica também o caminho sem template (POST sem template_id): orcamento_id preenchido, proposta_template_id NULL, campos_pendentes_revisao=[].
O smoke test scripts/smoke_test_modulos_criticos.py continua passando.
O capítulo manual/26_propostas.md ganha uma seção curta documentando o ciclo "Orçamento → Modelo → Proposta" com print do modal "Gerar Proposta" mostrando o <select> com pelo menos 1 template.
Out of scope
Mudar a lógica de gerar_proposta em si (já validada).
Refatorar propostas_consolidated.py (god-file).
Mexer no fluxo de aprovação da proposta pelo cliente (já coberto por tests/test_e2e_proposta_aprovacao_cliente.py).
Implementar UI nova para gestão de templates (a tela /propostas/templates já existe).
Steps
Auditar caminho atual no código — confirmar que propostas_consolidated.py expõe POST para criar template (/propostas/templates/novo) e que views/orcamentos_views.py expõe POST para criar orçamento e POST /<id>/gerar-proposta. Documentar parâmetros esperados.
Atualizar o seeder do demo Alfa — adicionar criação de 1 PropostaTemplate completo (padrao=True), de 1 orçamento adicional, e re-gerar 2 das 3 propostas a partir de orçamentos (uma com template, uma sem) para cobrir os dois caminhos. Idempotente (não duplica em re-execução).
Teste E2E automatizado — criar tests/test_e2e_orcamento_proposta_modelo.py com os 6 passos do "Done looks like". Usar o cliente de teste do Flask + login real, sem mocks.
Re-rodar o smoke — scripts/smoke_test_modulos_criticos.py deve continuar 100% verde após o seed atualizado.
Documentar no manual — adicionar seção "Ciclo Orçamento → Modelo → Proposta" em manual/26_propostas.md com o passo-a-passo da UI e um print do modal Gerar Proposta.
