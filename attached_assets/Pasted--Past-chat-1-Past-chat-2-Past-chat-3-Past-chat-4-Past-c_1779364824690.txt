## Diagnóstico da Task #15

A task faz sentido e está **parcialmente preparada**, mas ainda não completa.

Hoje:

1. A tela `templates/importacao/preview_fluxo.html` já tem `<select>` de categoria nas **saídas**, inclusive com `Categorias Personalizadas`. Porém a tabela de **Entradas** ainda só exibe dados de leitura: data, cliente, descrição, CC/obra, valor, status e fuzzy match. Não há `<select>` de categoria nas entradas. 

2. A rota `fluxo_caixa_confirmar` já possui a função `_aplicar_categoria(row, cat_val)`, que entende o padrão `cfc_<id>` e grava `categoria_fluxo_caixa_id` no dicionário da linha. Mas ela só é aplicada em `saidas_auto` e `saidas_manual`, não em `entradas`. 

3. No serviço `services/importacao_excel.py`, a criação de `ContaReceber` para entradas não recebe `categoria_fluxo_caixa_id`, e o `FluxoCaixa` criado quando a entrada está `PAGO` também não passa esse campo. 

4. Atenção: pelo trecho do `models.py`, `ContaReceber` **não aparenta ter** a coluna `categoria_fluxo_caixa_id`. Então a parte do “Done looks like” que diz “sets categoria_fluxo_caixa_id on the ContaReceber or FluxoCaixa entry record” deve ser interpretada com cuidado: hoje, pelo código visto, o caminho seguro é gravar no `FluxoCaixa`. Para gravar também em `ContaReceber`, precisa migration/model.

### 3. No confirmar, aplicar categoria também nas entradas

Depois dos loops das saídas, adicionar:

```python
# Aplicar categoria personalizada nas entradas
for i, row in enumerate(entradas):
    cat_entrada = request.form.get(f'cat_entrada_{i}')
    if cat_entrada:
        _aplicar_categoria(row, cat_entrada)
```

Ponto importante: a função `_aplicar_categoria` hoje não valida se a categoria pertence ao admin nem se é `ENTRADA`. Para ficar mais seguro, dá para criar uma versão validada:

```python
def _aplicar_categoria(row, cat_val, tipo_esperado=None):
    if not cat_val:
        return

    if cat_val.startswith('cfc_'):
        try:
            cat_id = int(cat_val[4:])
        except ValueError:
            return

        q = CategoriaFluxoCaixa.query.filter_by(
            id=cat_id,
            admin_id=admin_id,
            ativo=True
        )

        if tipo_esperado:
            q = q.filter_by(tipo=tipo_esperado)

        cat_obj = q.first()
        if not cat_obj:
            return

        row['categoria_fluxo_caixa_id'] = cat_obj.id
        row['tipo_categoria'] = row.get('tipo_categoria') or (
            'RECEITA' if tipo_esperado == 'ENTRADA' else 'OUTROS'
        )
    else:
        row['tipo_categoria'] = cat_val
        row['categoria_fluxo_caixa_id'] = None
```

Uso:

```python
_aplicar_categoria(row, cat_editada, tipo_esperado='SAIDA')
_aplicar_categoria(row, cat_manual, tipo_esperado='SAIDA')
_aplicar_categoria(row, cat_entrada, tipo_esperado='ENTRADA')
```

## Sobre gravar também em `ContaReceber`

Pelo trecho analisado, `ContaReceber` não tem `categoria_fluxo_caixa_id`. Então há duas opções:

### Opção A — mais segura e menor escopo

Gravar a categoria personalizada apenas no `FluxoCaixa`.

Vantagem: resolve o impacto prático da importação de fluxo, sem mexer no banco.

### Opção B — mais completa

Adicionar `categoria_fluxo_caixa_id` também em `ContaReceber`.

Aí precisa:

```python
categoria_fluxo_caixa_id = db.Column(
    db.Integer,
    db.ForeignKey('categoria_fluxo_caixa.id'),
    nullable=True
)
```

E migration:

```sql
ALTER TABLE conta_receber
ADD COLUMN categoria_fluxo_caixa_id INTEGER NULL;

ALTER TABLE conta_receber
ADD CONSTRAINT fk_conta_receber_categoria_fluxo_caixa
FOREIGN KEY (categoria_fluxo_caixa_id)
REFERENCES categoria_fluxo_caixa(id);
```

Depois, no `ContaReceber(...)`:

```python
categoria_fluxo_caixa_id=cfc_id,
```

Eu faria a **Opção A primeiro**, porque a task fala “ContaReceber or FluxoCaixa”, e o `FluxoCaixa` é o registro que realmente alimenta o fluxo.
