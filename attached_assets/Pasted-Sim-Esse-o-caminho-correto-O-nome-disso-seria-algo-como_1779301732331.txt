# A regra principal

Nem todo dropdown deve ser livremente editável.

Eu separaria assim:

## Dropdown editável pelo usuário

Pode ter CRUD completo.

Exemplos:

* origem do lead;
* tipo de obra;
* tipo de material;
* motivo de perda;
* categoria de fornecedor;
* categoria financeira;
* forma de pagamento;
* tipo de reembolso;
* tipo de ocorrência;
* unidade de medida.

## Dropdown de status de sistema

Tem que tomar cuidado.

Exemplos:

* status de proposta;
* status de obra;
* status de conta a pagar;
* status de medição;
* status de RDO.

Esses podem até ter **nome e cor editáveis**, mas o valor interno não deveria mudar.

Exemplo:

```text
valor interno: pendente
nome exibido: Aguardando aprovação
cor: amarelo
```

O código continua usando `pendente`, mas o usuário enxerga “Aguardando aprovação”.

Isso evita quebrar regra de negócio.

# Modelo ideal de banco

Eu criaria duas tabelas principais.

## Tabela 1 — Grupo de Dropdown

```python
class DropdownGrupo(db.Model):
    __tablename__ = 'dropdown_grupo'

    id = db.Column(db.Integer, primary_key=True)
    admin_id = db.Column(db.Integer, nullable=True)

    modulo = db.Column(db.String(80), nullable=False)
    slug = db.Column(db.String(120), nullable=False)
    nome = db.Column(db.String(150), nullable=False)
    descricao = db.Column(db.Text)

    tipo = db.Column(db.String(50), default='lista')
    editavel = db.Column(db.Boolean, default=True)
    sistema = db.Column(db.Boolean, default=False)

    ativo = db.Column(db.Boolean, default=True)
    ordem = db.Column(db.Integer, default=0)
```

## Tabela 2 — Opções do Dropdown

```python
class DropdownOpcao(db.Model):
    __tablename__ = 'dropdown_opcao'

    id = db.Column(db.Integer, primary_key=True)
    grupo_id = db.Column(db.Integer, db.ForeignKey('dropdown_grupo.id'), nullable=False)
    admin_id = db.Column(db.Integer, nullable=True)

    valor = db.Column(db.String(120), nullable=False)
    nome = db.Column(db.String(150), nullable=False)

    descricao = db.Column(db.Text)
    cor = db.Column(db.String(20))
```

# Como usar nos formulários

Em vez de colocar isso no HTML:

```html
<option value="Instagram">Instagram</option>
<option value="Indicação">Indicação</option>
<option value="Google">Google</option>
```

Você passaria opções pelo backend:

```python
origens = get_dropdown_options('crm_origem')
tipos_obra = get_dropdown_options('crm_tipo_obra')
motivos_perda = get_dropdown_options('crm_motivo_perda')
```

E no template:

```html
<select name="origem_id" class="form-select">
  {% for item in origens %}
    <option value="{{ item.id }}">{{ item.nome }}</option>
  {% endfor %}
</select>
```

Para campos que usam string:

```html
<select name="tipo_ocorrencia" class="form-select">
  {% for item in tipos_ocorrencia %}
    <option value="{{ item.valor }}">{{ item.nome }}</option>
  {% endfor %}
</select>
```

# Ponto importante: não deletar opções usadas

Se uma opção já foi usada em lead, proposta, financeiro ou obra, ela não deve ser excluída fisicamente.

Ela deve ser **desativada**.

Exemplo:

```text
Origem: Facebook Ads
```

Se existem 200 leads com essa origem, o usuário não pode deletar e quebrar histórico.

O certo é:

```text
ativo = False
```

Ela some dos novos formulários, mas continua aparecendo nos registros antigos.

---

# Como evitar hardcode de verdade

Você precisa criar uma regra técnica:

> Todo campo `<select>` com opções de negócio deve usar uma lista mestra cadastrável.

Exceções:

* opções técnicas internas;
* status com lógica fixa;
* campos booleanos;
* enums que controlam regra crítica.

Exemplo de regra:

| Tipo de dropdown            | Pode virar CRUD? | Observação                              |
| --------------------------- | ---------------: | --------------------------------------- |
| Origem do lead              |              Sim | Lista mestra normal                     |
| Tipo de obra                |              Sim | Lista mestra normal                     |
| Motivo de perda             |              Sim | Lista mestra normal                     |
| Categoria de fornecedor     |              Sim | Lista mestra normal                     |
| Forma de pagamento          |              Sim | Lista mestra normal                     |
| Status “pago/pendente”      |          Parcial | Valor interno fixo, rótulo/cor editável |
| Sim/Não                     |              Não | Campo booleano                          |
| Tipo de usuário admin/comum |              Não | Regra de permissão                      |
| Módulo do sistema           |              Não | Estrutural                              |

# Veredito

Sua ideia é mais importante do que parece.

Isso resolveria três problemas grandes:

1. **Usabilidade:** o usuário sabe onde cadastrar cada coisa.
2. **Manutenção:** o programador para de espalhar opções hardcoded.
3. **Escalabilidade:** novos módulos passam a usar o mesmo padrão de listas mestras.

Eu chamaria essa task de:

## **Task — Central Universal de Cadastros, Catálogos e Dropdowns**

E classificaria como:

| Critério           | Avaliação  |
| ------------------ | ---------- |
| Valor para usuário | Muito alto |
| Valor técnico      | Muito alto |
| Risco              | Médio      |
| Esforço inicial    | Médio      |
| Esforço completo   | Alto       |
| Prioridade         | Alta       |

Essa task deve entrar antes de várias outras, porque ela organiza a base do sistema.
Sem isso, cada nova funcionalidade vai continuar criando dropdown próprio, tabela própria ou opção fixa no HTML.
