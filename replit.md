## Overview
SIGE v9.0 (Sistema de Gestão Empresarial) is a multi-tenant business management system for SMBs, designed to automate and streamline critical operations. It covers commercial proposal generation, employee and payroll management, construction project control (Daily Work Reports - RDO), and comprehensive financial calculations. The system aims to enhance operational efficiency, provide end-to-end oversight from sales to project execution, and improve overall business management.

## User Preferences
- Priorizar soluções automáticas que funcionem no deploy
- Evitar intervenção manual no banco de produção
- Implementar logs detalhados para debugging
- Sistema deve ser resiliente a diferenças entre ambientes
- Interface moderna com cards elegantes em vez de listas simples
- Design limpo e profissional com gradientes e animações suaves
- Template unificado em todas as páginas do sistema
- Contraste adequado de texto (valores em preto) para melhor legibilidade
- Proteção robusta contra erros de transação de banco de dados
- Scripts automatizados para deploy de correções críticas em produção
- Código de produção limpo, separado de testes para performance otimizada
- Testes dedicados estruturados para validação contínua dos módulos consolidados
- Links RDO devem apontar para rota moderna `/funcionario/rdo/consolidado` com funcionalidades completas
- Ambiente de produção com 80 tabelas deve ser preservado durante migrações
- Operações destrutivas (exclusão) devem usar POST via formulários JavaScript
- Evitar auto-fill de campos que possam interferir em filtros de busca

## System Architecture
The system utilizes a Flask backend with SQLAlchemy ORM, a PostgreSQL database, and Jinja2 for server-side templating. Bootstrap is used for the frontend, and Docker for deployment. The codebase is structured with a `views/` Python package for domain-specific modules, all using a unified `base_completo.html` template.

**Key Architectural Decisions & Features:**
-   **Multi-tenancy and Security:** Features data isolation, role-based access control, CSRF protection, restricted CORS, rate limiting, and secure key handling.
-   **Event-Driven Architecture:** Employs an `EventManager` (Observer Pattern) for automated cross-module data flow.
-   **UI/UX Design:** Focuses on modern, responsive layouts with modular components, a cohesive color palette, hierarchical typography, consistent spacing, WCAG accessibility, and Mobile-First Design principles, including elegant cards, gradients, and smooth animations.
-   **Data Integrity & Management:** Includes automated database migrations, atomic transactions via `safe_db_operation`, and automated error diagnostics for SQLAlchemy.
-   **Dynamic Content Generation:** Supports dynamic PDF generation with custom headers and company customization through logo uploads and configurable color schemes.
-   **Core Modules:**
    -   **Commercial Management:** Proposal management with templates, automated calculations, real-time recalculation, client portal access, service catalog, and parametric budgeting.
    -   **Employee & Payroll Management:** Employee registration, automated time clocking (GPS, facial recognition, geofencing), flexible schedules, automated CLT-compliant payroll, dynamic PDF holerite generation, and accounting integration.
    -   **Construction Project Management:** Daily Work Reports (RDO) with dynamic service progress, consolidated routing, mobile photo uploads, and integration with Cronograma V2; Biweekly Project Measurement. When the same employee is assigned to N activities (subatividades + cronograma tasks) in one RDO, the input hours are normalized via `utils/rdo_horas.normalizar_horas_funcionario` (max hours / N distinct activities) so the daily total per employee never exceeds the worked shift; the visualization aggregates rows by employee, showing one card per person with the list of activities and per-activity hours. Legacy save endpoints (`POST /rdo/criar` in `views/rdo.py`, `POST /rdo/salvar` via `rdo_salvar_unificado`, and `rdo_crud.salvar_rdo` in `crud_rdo_completo.py`) also pass through the same helper and emit a `[LEGACY-RDO]` warning log used as production telemetry to detect any remaining external caller before they can be removed (Task #9).
    -   **Financial & Accounting:** Double-entry bookkeeping, automated journal entries, chart of accounts, trial balance, DRE, and enhanced FluxoCaixa.
    -   **Logistics & Inventory:** Warehouse management with traceability, full CRUD for suppliers, material flow, batch/lot selection, serialized item transitions, and optimistic locking; Material Purchase Management with dynamic item tables and automated cost registration.
    -   **Fleet Management:** Vehicle management, expense tracking, TCO dashboards, and critical alerts.
    -   **Food Management:** Mobile-first interface for restaurant and food entries with dynamic multi-item launching and automatic cost calculation.
    -   **Client Portal:** Token-based access for clients to view project progress, approve purchases, and upload payment receipts. The construction-site portal (`/portal/obra/<token>` and `/portal/obra/<token>/rdo/<id>`) was refactored on top of a shared `templates/portal/_base.html` layout: design tokens via CSS variables, neutral header with company logo/name + obra/cliente, brand band injecting the tenant's `cor_primaria` as `--brand`, hero progress ring + KPI grid, modern timeline cronograma, compras (pendentes/resolvidas) with comprovante upload, V2 bidding maps preserving JS-critical class hooks (`cotacao-cell`, `cotacao-radio`, `forn-header`, `cell-selected`, `cell-min-val`, `cell-min-prazo`, `radio-visual`), medições, RDOs with thumbnails, and a foto modal on the RDO detail. When `portal_ativo=False`, the portal returns a branded "Portal temporariamente inativo" page (200) instead of a generic 404, while POST action routes still 404 to prevent mutations.
    -   **Competitive Bidding:** Multi-supplier comparison tool with an N items × N suppliers grid, inline editing, and interactive client selection.
    -   **Automated Project Scheduling (Cronograma V2):** MS-Project-style Gantt chart with task hierarchy and drag-and-drop. Automatic schedule creation from service templates upon proposal approval.
    -   **Configurable Proposal Clauses:** Dynamic clause management per proposal and template, allowing structured, orderable entries.
    -   **Responsible Engineer Management:** Dedicated `EngenheiroResponsavel` model supporting per-proposal overrides and fallback mechanisms.
    -   **Project-Client Linking:** `Obra.cliente_id` FK to `Cliente` for improved data integrity, with fallback to legacy text fields.
    -   **Employee Metrics:** Unified service for calculating employee KPIs and costs for salarists and day-workers.
    -   **Configurable Theme System:** Per-tenant visual themes stored in `ConfiguracaoEmpresa`, allowing selection from presets or custom color adjustments (primary, secondary, header/nav, app background) overriding Bootstrap via CSS variables.
    -   **Resilient Client Approval Flow:** Robustness fixes for proposal approval, handling null proposal subjects and creating synthetic `Cliente` records for legacy data to prevent failures in `Obra` creation.
    -   **Searchable Supplier Dropdown on /compras/nova:** Implemented Select2 for supplier and project search, providing typeahead functionality. Includes multi-tenant validation to prevent cross-tenant data leakage.
    -   **Single Proposta Model:** Consolidated proposal data into a single `Proposta` model, dropping several legacy tables after data backup.
    -   **Initial Schedule Review on Obra:** Implemented a two-state visit gate for newly created `Obra` instances, requiring administrative review of the initial schedule before full project access. Includes a reset mechanism and backfill for existing production data.
    -   **Expandable Service Composition (Task #18):** Each service in the budget/proposal renders as a collapsible Bootstrap card via the partial `templates/propostas/_servico_card.html`. Collapsed view shows quantity × unit, unit price, total, and pills with sale subtotals per category (Mão de obra/Material/Equipamento — empty categories hidden). Expanded view shows three columns (one per category) with each input listed (qty × unit price × total) plus "Itens inclusos / Itens exclusos" bullet lists. Helper `composicao_venda_agrupada(snapshot, custo_unit, preco_venda_unit, quantidade)` in `services/orcamento_view_service.py` converts the cost snapshot into sale values via proportional markup ratio (preserving the cost mix); when cost is zero but sale exists, sale is distributed equally among inputs with coefficient > 0 to keep the composition visible. New `OrcamentoItem.itens_inclusos` / `itens_exclusos` text fields are propagated into `PropostaItem` on `gerar_proposta` and `duplicar`. Footer shows arithmetic mean of the per-service unit prices across services (simple, not weighted — units vary across items). Applied to `templates/orcamentos/editar.html`, `templates/propostas/detalhes_proposta.html` (internal proposal view), and `templates/propostas/portal_cliente.html` (public portal). Shared CSS in `static/css/servico_card.css`, including a mobile override that restores `display: table` for tables inside `.servico-card .table-responsive` (the portal hides regular tables on mobile).
    -   **Catálogos — Importação por Excel (Task #23):** Botões "Baixar modelo" e "Importar Excel" nas telas de Catálogo de Insumos (`/catalogo/insumos`) e Templates de Cronograma (`/cronograma/templates`). Service compartilhado `services/catalogo_excel.py` gera modelos `.xlsx` (com aba "Instruções") usando openpyxl e processa imports linha-a-linha com lista de rejected, escopados por `admin_id` (multi-tenant). Insumos: upsert por `(admin_id, nome)` case-insensitive; `preco_base` cria nova vigência via helper que espelha o comportamento de `catalogo.insumo_novo_preco` e dispara `recalcular_servico_preco` para serviços afetados. Templates de Cronograma: upsert por `(admin_id, nome)`; itens existentes são substituídos pelos da planilha; `parent_ordem` (referência à `ordem` de outra linha) resolvido via topo-sort; ciclos são detectados em DRY-RUN ANTES de tocar no banco (template existente nunca fica vazio por hierarquia inválida). Proteções de upload: limite de 5 MB por arquivo (mitiga zip-bomb/OOM), teto de 5000 linhas de dados por aba, e `load_workbook(read_only=True, data_only=True)` para baixo consumo de memória. Rotas registradas em `views/catalogo_views.py` (`/insumos/modelo-excel`, `/insumos/importar-excel`) e `cronograma_views.py` (`/templates/modelo-excel`, `/templates/importar-excel`); UI usa modal Bootstrap com `enctype="multipart/form-data"` e CSRF token.
    -   **RDO — peso da tarefa principal por funcionário (Task #38):** Quando o mesmo funcionário aparece em ≥ 2 atividades no RDO do dia, o apontador pode marcar uma das tarefas como **principal** com um peso livre (0–100%) num modal "Distribuição de horas" (`templates/rdo/editar_rdo.html`) que intercepta o submit em fase de captura. O peso por linha é persistido em `RDOMaoObra.peso_distribuicao` (Integer NULL — migration 145). O helper `utils.rdo_horas.normalizar_horas_funcionario` ganhou parâmetro `pesos={(func_id, key): int}`: linhas com peso recebem `peso/100` da jornada-base; linhas NULL dividem o restante (`100 - sum_pesos`) igualmente. Sem peso para nenhuma linha → divisão igual histórica preservada (zero regressão). Backend (`rdo_editar_sistema.salvar_edicao_rdo`) lê hidden fields `peso_dist_sub_<sub_mestre_id>_func_<func_id>` enviados pelo modal e passa o dict ao helper. Visualização (`templates/rdo/visualizar_rdo_moderno.html`) mostra badges "principal X%" / "secundária Y%" ao lado de cada atividade do funcionário. Cobertura pytest: `tests/utils/test_rdo_horas.py` (13 casos — todos os cenários da PRD + clamp + isolamento entre funcionários).
    -   **CRM de Leads — Kanban + listas mestras (Task #42):** Novo módulo `CRM` no menu superior com submenus Kanban / Lista / Novo Lead / Cadastros do CRM. Blueprint `crm_views.py` (prefix `/crm`) registrado em `app.py`. Modelos em `models.py`: `LeadStatus` enum (8 estágios fixos: Em fila → Em andamento → Enviado → Validação → Aprovado → Feedback → Congelado → Perdido), 7 listas mestras com `UNIQUE(admin_id, nome)` + flag `ativo` (`CrmResponsavel`/`CrmOrigem`/`CrmCadencia`/`CrmSituacao`/`CrmTipoMaterial`/`CrmTipoObra`/`CrmMotivoPerda`), `Lead` (todos campos da planilha do cliente + FKs opcionais para `Cliente`/`Proposta`/`Obra`) e `LeadHistorico` (timeline `ON DELETE CASCADE` com `admin_id` exigido pelo auto-fix universal). Migration **146** (`migration_146_crm_leads_e_seed_generico` em `migrations.py`) cria as 9 tabelas via `CREATE IF NOT EXISTS` + chama `crm_seeds.seed_listas_mestras_para_todos_admins()` que popula as 6 listas NÃO-NOMINAIS (Origens/Cadências/Situações/Tipos de Material/Tipos de Obra/Motivos de Perda) com valores genéricos para todo admin existente — **idempotente por lista** (não duplica e não reaparece se admin apagou). A lista `crm_responsavel` **nunca recebe seed** — cada admin cadastra o time. Kanban (`templates/crm/kanban.html`) usa Sortable.js (CDN 1.15.0) para drag-and-drop com endpoint `POST /crm/<id>/mudar_status` (JSON) que: (a) atualiza `status_changed_at`, (b) registra `LeadHistorico`, (c) auto-preenche `data_envio` quando vai para "Enviado", (d) **bloqueia (HTTP 422 + flag `requer_motivo`)** mover para "Perdido" sem `motivo_perda_id` (frontend abre modal com select). Filtros: busca textual, responsável, origem, tipo de obra, mês de chegada. Cards mostram badge "parado há Xd" (warning ≥7d, danger ≥14d). Helper `vincular_ou_criar_cliente(lead, admin_id)` em `crm_views.py` busca cliente por telefone normalizado (`re.sub(r'\D','')`) ou e-mail case-insensitive dentro do tenant; se achar, vincula e completa campos vazios; se não, cria novo `Cliente`. Tela de Cadastros (`templates/crm/cadastros.html`) é tabbed com 7 listas, suporta criar/renomear/ativar-desativar/excluir. Excluir item em uso é **bloqueado** com mensagem orientando a usar "Desativar" (preserva integridade referencial dos leads históricos). Multi-tenant: query base filtra por `admin_id`; funcionário comum só vê leads onde `CrmResponsavel.nome == current_user.nome` (heurística porque `CrmResponsavel` é cadastro livre, sem FK obrigatória para `Usuario`). Apenas ADMIN/SUPER_ADMIN/GESTOR_EQUIPES acessam Cadastros e podem excluir leads. CSRF protege todos os POSTs (form ou header `X-CSRFToken`).
    -   **Concluir/Reativar Obra (Task #17):** Mirrors the `Funcionario.ativo` pattern for `Obra` — the existing `Obra.ativo` boolean now drives a "Conclude / Reactivate" UX without ever deleting historical data. The listing in `templates/obras_moderno.html` splits into "Obras Ativas" and "Obras Concluídas / Inativas" sections (with a separator and muted styling for the inactive group). Active cards expose a "Concluir" action and inactive cards expose only "Detalhes" + "Reativar". Both actions hit a new tenant-isolated JSON endpoint `POST /api/obra/<id>/toggle-ativo` (mirrors `toggle_funcionario_ativo`) after a JS `confirm()`. The legacy form-based `POST /obras/toggle-status/<id>` (still used by `templates/obras/detalhes_obra_profissional.html`) was hardened with `admin_id` filtering. The edit form (`templates/obra_form.html`) gained an "Obra ativa" checkbox; `views/obras.py:editar_obra` reads `'ativo' in request.form`. Inactive obras remain reachable via `detalhes_obra` and stay hidden from selectors that already filter `Obra.ativo == True` (financeiro, RDO, ponto, equipe, dashboards).

## Replit Environment Notes
- **Runtime:** Python 3.11 with gunicorn served via `.pythonlibs/bin/gunicorn`
- **Database:** Replit built-in PostgreSQL (via `DATABASE_URL` pointing to `helium`)
- **Auth:** Custom Flask-Login (no external auth provider)
- **Session Secret:** Stored in `SESSION_SECRET` environment variable
- **Run command:** `.pythonlibs/bin/gunicorn --bind 0.0.0.0:5000 --reuse-port --reload main:app`
- **Added models:** `ManutencaoVeiculo` and `RdoAtividade` (legacy stub) to fix missing FK targets in `models.py`
