Architecture Review · deepening opportunities

Classificação de Fluxo de Caixa

Avaliação da arquitetura planejada antes de implementar. Linguagem de domínio do CONTEXT.md; respeita o ADR-0002. Foco: fronteiras de módulo do novo classificador, testabilidade isolada do Flask, e o risco de inchar processar().

Fatos: processar() 325 linhas / ~8 queries · importar() 413 linhas · preview = HMAC round-trip sem estado server-side · decisão auto/Pendente em variável local.

1 · Separar Parse de Classify

Strong

Módulos: services/importacao_excel.py › ImportacaoFluxoCaixa.processar

Problema. processar() é um módulo shallow: interface minúscula (arquivo, período) escondendo 7 responsabilidades — abrir workbook, carregar 7 entidades do DB, parsear colunas por índice mágico (row[10..13] caçando o valor), filtrar período, classificar em 3 níveis, pré-computar "apenas pagamento", resolver categoria nomeada. O plano adiciona carregar regras/correções, gerar a fila, manter linhas vivas — empurrando pra ~380 linhas. Toda a complexidade num lugar sem costura interna.

Solução. Dois módulos profundos atrás de interfaces pequenas: um Leitor de Planilha (workbook + período → list[Lançamento], sem DB, sem classificação) e um Classificador (candidato 2). processar() vira cola fina: carrega contexto, chama os dois.

Antes — god-method

processar() · 325→380 linhas

abrir workbook7 queries de entidade parse colunas (índice mágico)filtro período classificar 3 níveisdecisão auto/Pendente apenas-pagamentocategoria nomeada + regras/correções+ gerar fila

Teste = subir Excel real + app context. Sem teste unitário do parse nem da decisão.

Depois — 3 módulos profundos

Leitor de Planilha
ler(wb, periodo) → [Lançamento] · só openpyxl
Classificador
classificar_lote(lançamentos, ctx) → Resultado
processar() · ~40 linhas
cola: carrega contexto, chama Leitor + Classificador

Parse testável com um .xlsx minúsculo, sem DB. Decisão testável sem openpyxl.

Deletion test. Apague o Leitor: o índice mágico das colunas (row[10..13], offset do cabeçalho da aba Saída) re-aparece espalhado em quem parseia. Concentra complexidade → ganha o sustento. Leverage: o parse vira o lugar único onde "o formato Veks" é conhecido.

2 · Um Classificador que decide o veredito, não só devolve categoria

Strong · top pick

Módulos: planejado services/classificador_cadastro.py · hoje espalhado em processar() + 8 funções de nível de módulo

Problema. Hoje a ordem (plano → reembolso → keywords) e a decisão "auto vs Pendente" vivem numa variável local precisa_revisao dentro de processar() — não há locality. As funções puras (_classificar_keywords etc.) são testáveis isoladas, mas o bug mora em como são orquestradas. O plano cria classificar_por_cadastro(...) → categoria | None — uma função shallow: devolve categoria-ou-None e deixa processar() re-interpretar o None como Pendente e aplicar a ordem Regra→Memória Exata. A dispersão se repete.

Solução. Tornar o Classificador deep: a interface devolve um Veredito completo, e a ordem de resolução (Regra de Classificação → Memória Exata → Pendente) e o desempate por prioridade vivem dentro dele. processar() não decide nada de classificação — só consome o veredito.

Antes — função rasa + orquestração externa

cat = classificar_por_cadastro(...)   # → str | None
# processar() ainda precisa:
if cat is None:
    cat = memoria_exata.get(texto)     # ordem aqui
if cat in (None,'Outras Saídas'):
    pendentes.append(...)              # veredito aqui
else:
    auto.append(...)

Test surface = funções de keyword. A decisão real (Pendente?) não tem teste próprio.

Depois — veredito é a interface

v = classificador.classificar(lanc, ctx)
# v.categoria_id, v.categoria_nome
# v.origem_decisao  (regra|memoria|fallback)
# v.eh_pendente     ← o veredito
(auto if not v.eh_pendente else pendentes).append(...)

Test surface = o veredito. assert classificar(x).eh_pendente is True. A ordem Regra→Memória→Pendente tem locality.

Ordem de resolução (dentro do módulo, com locality):

flowchart LR
  L[Lançamento] --> R{Regra de Classificação?\nmenor prioridade}
  R -->|casa| C1[Veredito: categoria]
  R -->|nenhuma| M{Memória Exata?\ntexto idêntico}
  M -->|sim| C2[Veredito: categoria]
  M -->|não| P[Veredito: eh_pendente]

Deletion test. Apague o Classificador: a ordem de resolução e a regra auto/Pendente re-aparecem em processar() e em todo endpoint do loop ao vivo (classificar-termo, corrigir-linha) que precisa reclassificar. Concentra complexidade em 3+ chamadores → ganha o sustento, com folga. É o módulo de maior leverage do projeto.

3 · Onde mora o estado da Sessão de Preview?

Worth exploring

Módulos: importacao_views.py (_assinar_payload/ _verificar_payload L36-69) · plano Passo 13

Problema. Hoje o preview é stateless: o payload vai assinado (HMAC) e round-trip pelo navegador — não há estado server-side. Mas o loop ao vivo (drill-down + reclassificar a cada Termo classificado, sem re-upload) precisa dos Lançamentos individuais vivos entre requests. O plano só acena: "token assinado + cache server-side". Essa é a costura mais subespecificada — se não for decidida, vira estado ad-hoc (session global, dict de módulo) vazando em importacao_views.

Duas adaptadores possíveis (= seam real só com 2):

A · Payload-como-estado

Cada ação re-POSTa o payload assinado; servidor reclassifica em memória e devolve o delta. Zero infra nova, idêntico ao que confirmar já faz.

Custo: ~3700 Lançamentos num campo oculto re-enviado a cada clique.

B · Módulo Sessão de Preview

Store server-side token → [Lançamento] + estado da fila, interface carregar/reclassificar/confirmar. Requests pequenos.

Custo: estado mutável + ciclo de vida/expiração pra gerir.

Decisão pendente. O plano precisa escolher explicitamente A ou B antes da Fase F. Recomendo grelhar: o volume (3700 linhas) e a frequência de cliques decidem. Não contradiz nenhum ADR — é um seam novo que o ADR-0002 não cobre.

4 · A fan-out de importar()

Speculative

Problema. importar() tem 413 linhas: dedup, auto-criação de fornecedor, fan-out GestaoCustoPai/Filho + FluxoCaixa + ContaPagar, entradas, relatório. É god-method — mas pré-existente e fora do escopo desta feature.

Recomendação. Não mexer agora (o skill evita refactor não relacionado). A feature só toca importar() pra passar categoria_fluxo_caixa_id — mudança cirúrgica. Fica registrado como dívida adjacente para um ciclo futuro.

★ Por onde eu começaria

Candidato 2 primeiro (o Classificador que devolve Veredito). É o módulo que a feature cria de qualquer jeito — acertar a interface (devolve veredito + dona da ordem de resolução) é a decisão de maior leverage e ataca direto os dois objetivos: testabilidade (test surface = o veredito) e não inchar processar() (a decisão sai de lá). O Candidato 1 (separar Parse) cai naturalmente depois. O Candidato 3 (Sessão de Preview) é uma decisão a grelhar antes da Fase F, não um refactor agora.
